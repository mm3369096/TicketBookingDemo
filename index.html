與大家分享的專題提案：為什麼想選擇「搶票+商城」呢？

1. 讓架構變簡單，把時間留給更核心的技術學習

其實「寵物 Airbnb」是個很棒的點子，但因為日期重疊和表單驗證會花去我們非常多時間，所以想提議改成單純的座位模型（只有空位、鎖定、售出三種狀態）。這樣能大大減輕大家在資料庫 (SSMS) 上的壓力，讓我們把寶貴的專題時間，投資在面試官最看重的「高併發處理」與「資料庫事務」上，對大家的求職會更有幫助喔。

2. 照顧彼此的進度：我們的三階段安心方案

知道大家可能會擔心搶票系統太難寫不出來，別擔心，我們有安排很有彈性的備案：

  • Level 1 (安心保底)：用標準的 SQL Update 加上 Security，這樣我們一定能交出完整的作品。
  • Level 2 (順利達標)：嘗試加上 Hibernate 樂觀鎖 (@Version)，解決基本的超賣問題。
  • Level 3 (行有餘力再加分)：如果大家進度順利，再來挑戰 Redis 預扣庫存跟排隊室，一起讓專題變得更厲害！

3. 讓每個人都能發揮所長的分工建議

這個提案最大的好處是，各個功能彼此獨立，大家可以依照自己的興趣和步調來開發，不用怕卡到別人的進度：

  • 商城與訂單 (CRUD)：適合喜歡穩紮穩打，把基礎練熟的同學。
  • 搶票與併發 (Redis/Locking)：適合想挑戰高難度技術、研究效能優化的同學。
  • 視覺化選位 (Vue/SVG)：適合對前端 Vue 或 UI 設計有熱忱的同學。
  • B2B 管理後台：能練習複雜的 SQL 統計，適合想深入資料處理的同學。

[備註小語] 這個 Prototype 是先用 Vue 3 單檔寫出來的，主要想讓大家對於「資料結構精簡化」有一個比較具體的想像，後續我們實作時再一起轉移到 Spring Boot 專案裡就好囉。
